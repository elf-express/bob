# BOB 架構原則:共用的殼 + 資料跟著專案走

> 類型:架構決策(Architecture Decision)
> 日期:2026-07-03
> 狀態:草案 — 待使用者 review
> 適用:`@elf-express/bob`(BOB 自 team-project-template 抽出的獨立工具 repo)

---

## 一、背景與問題

BOB 已從 team-project-template 抽成獨立 repo(`@elf-express/bob`),定位為「團隊共用的開發輔助工具」。抽出後浮現一個核心問題:

> **一份共用的 BOB,怎麼服務多個專案而不讓資料互相污染?**

### 現況(code 實證)

BOB 目前把所有狀態存在**工具目錄**的單一 sqlite(`data/bob-tools.sqlite`):

| code 位置 | 行為 | 後果 |
|---|---|---|
| `migrations/001_init.sql`:`project_binding CHECK (id = 1)` | 只能存一筆 binding | 一次只綁一個專案 |
| `scan-repository.js`:掃描先 `DELETE FROM commands_snapshot / tree_snapshot` 再寫 | 每次掃描全清重寫 | 切專案 → 前一個專案的快照被清空 |
| `file_tags`:以 `rel_path` 為 UNIQUE key | 無專案維度 | 不同專案同路徑的標註互蓋 |

**結論**:同一份 BOB 服務多個專案時,後掃的會覆蓋前面的。資料綁在「工具」上,不是綁在「專案」上。

---

## 二、架構原則(本 spec 要釘死的)

BOB 分成兩層,各走各的路:

### 1. 程式碼(工具本身)= 共用一套

- 一份 `@elf-express/bob`,各專案透過 npm devDependency(或 submodule)關聯。
- 只有**版本升級**才更新它 —— 一處維護、所有專案受益。

### 2. 資料(掃描 / 標註 / 結構說明)= 跟著專案走

- 每個專案的 BOB 資料**存在該專案自己裡**,不存在工具目錄。
- 資料屬於「當下服務的那個專案」:爬蟲的資料在爬蟲裡、模板的資料在模板裡。

### 心智模型

BOB = **VS Code / ESLint / Playwright 模式**:工具裝一套共用,每個專案的設定與內容長在各專案自己裡。升級工具不動任何專案的內容;內容 per-project,與「程式碼共用」天生不衝突。

---

## 三、現況差距 → 目標

| | 現況 | 目標 |
|---|---|---|
| 資料存哪 | 工具目錄 `data/bob-tools.sqlite` | 當下服務的專案內(如 `<專案>/.bob/`) |
| 多專案 | 切換即覆蓋 | 各專案各一份,怎麼切都不覆蓋 |
| 資料歸屬 | 綁工具 | 綁專案 |

---

## 四、本 spec 範圍邊界(明確劃清)

**本 spec 只釘「架構原則 / 方向」。** 以下實作細節**留待後續各自的實作 spec**,不在此決定:

- 資料在專案內的**確切路徑**(`.bob/` 還是別的)
- **存儲格式**:sqlite vs 文字檔(每檔一個 `.md` / `.json`)—— 直接影響能否進 git
- per-project 資料**是否進專案 git**(團隊共享 vs 本地個人)
- schema / migration 的具體改造、既有資料搬遷步驟

---

## 五、與「參考手冊」功能的關係

「參考手冊」(指令生命週期表等)是**全域內容** —— 所有專案一樣,屬於「共用的殼」的一部分,**不是 per-project 資料**。因此它不受本架構影響,可獨立實作,直接放進共用的 BOB code 即可。

---

## 六、衍生待辦(依此原則,各自獨立)

1. **[大]** 資料搬遷架構:把 db 從工具目錄搬進專案,決定路徑 / 格式 / 是否進 git。
2. **[中]** 參考手冊功能:全域內容,獨立於上,可先做。
3. **[小]** daemon(`bob-daemon.mjs`)搬進 repo(v1 未含,路徑硬編碼待重寫)。
